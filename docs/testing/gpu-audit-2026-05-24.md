# GPU Test Tier-Classification Audit — 2026-05-24 (GTX 1080)

Per-test tier classification for every GPU-gated test, calibrated on a real
GTX 1080. Companion to [`gpu-test-policy.md`](gpu-test-policy.md) (the policy)
and [`manual-gpu-pass-2026-05-24-gtx1080.md`](manual-gpu-pass-2026-05-24-gtx1080.md)
(the §4.3 hard-gate + Phase-0 GC trace).

Hardware: **NVIDIA GeForce GTX 1080**, compute capability 6.1 (sm_61), 8 GB
VRAM (~7.5 GB usable), driver via WSL2. Torch **2.7.1+cu118** (the `gpu-pascal`
uv extra; the default cu130 wheel ships no sm_61 cubin). bf16 is emulated on
Pascal and coerced to float16 by `coerce_dtype_for_capability`.

## Method

- Each file run in its **own pytest process** (`uv run --extra gpu-pascal
  --extra dev python -m pytest <file>`), `PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True`.
  Real-model tests each load the full ~3.3 GB checkpoint; per-process isolation
  frees it between files.
- Peak VRAM captured in-process via `torch.cuda.max_memory_allocated()`.
- "Move-to-CPU" criterion: a test earns a GPU tier only if it needs real
  SAM 3.1 weights, real CUDA kernels, or bitsandbytes quant. **The move-to-CPU
  set is empty** — every GPU-tagged test asserts on real-weight/real-kernel
  behavior a CPU stub cannot reproduce.

## Tier definitions (see policy doc)

- **gpu_local** — fits the GTX 1080 (≤ ~7 GB usable, CC 6.0+, NF4 + float16).
  Forward / inspection only; no training. Run on the dev box via
  `bash scripts/run_gpu_tests.sh local` (one process per file).
- **gpu_t4** — needs > 8 GB and ≤ 16 GB, **or** bf16-representative numerics,
  **or** a real training loop. Colab T4. **Provisional pending #137** (8 GB
  QLoRA training without gradient checkpointing — gradient checkpointing is
  abandoned, see the Phase-0 trace in `manual-gpu-pass-2026-05-24-gtx1080.md`).
- **gpu_xl** — beyond a T4 (> 16 GB / larger arch). Cloud auto-provision (#124).
  Currently empty.

## Bugs surfaced by the clean 1080 runs (fixed in this PR, commit 58c59a0)

These three gpu_local tests failed on the first clean real-GPU run for reasons
unrelated to VRAM (peaks 0.4–0.6 GB). All are product bugs that CPU tests — all
float32 — never exercised. Fixed so the gpu_local tier is genuinely green:

1. **G1 `test_channel_adapter_gpu::test_G1_real_forward_nchannel`** —
   `channel_adapter` Conv2d is built float32 and `_apply_dtype` casts only the
   inner model, so a bf16/fp16 forward raised
   `Input type (BFloat16) and bias type (float)`. Fix: cast the adapter to the
   inner model's dtype at construction + cast forward input to the adapter
   weight dtype (`models/sam3.py`).
2. **G4 `test_predict_nchannel_gpu::test_G4_real_nchannel_predict`** —
   `_row_outputs` did `v[r:r+1]` over **all** keys, but sam3
   `forward_grounding` returns non-tensor entries (`prev_encoder_out` dict,
   `encoder_hidden_states`) → `KeyError(slice)`. Fix: skip non-tensor values
   (`eval/evaluator.py`).
3. **base `test_gpu_predict::test_predict_base_model_cuda`** —
   `_BUILTIN_DEFAULT_IMAGE_SIZE` was 1024 but `load_sam31` uses 1008, so a
   config-less predict resized to 1024 and tripped sam3's RoPE
   `freqs_cis.shape == (H, W)` assertion (dtype-independent; would fail on any
   GPU). Fix: default to 1008 (`predict/runner.py`).

**VRAM-cleanup fixture:** once G1 actually completes a forward, running the
three `test_channel_adapter_gpu` tests in one process accumulated to a 7.69 GB
peak and OOM'd at G3's second model load. An autouse `_free_cuda_after_gpu_test`
fixture (`tests/conftest.py`, gated on `requires_compatible_gpu`) frees the CUDA
cache after each GPU test → file peak dropped to **5.94 GB**, all three pass.
