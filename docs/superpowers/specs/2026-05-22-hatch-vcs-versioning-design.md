# Switch versioning to hatch-vcs (tag-as-source-of-truth)

Resolves [#103](https://github.com/NguyenJus/custom-sam-peft/issues/103).

## Problem

`pyproject.toml` currently hard-codes `version = "0.8.0"`. With multiple
feature branches in flight, two branches can independently compute the same
"next" semver against the latest tag and stamp it into the manifest. Whichever
branch merges second carries a stale version that conflicts with the first
branch's tag.

The intent has always been that the git tag is authoritative, but today the
manifest is — the tag merely confirms what the manifest already declared.

A second symptom of the dual-source-of-truth problem already exists in the
tree: `src/custom_sam_peft/__init__.py` carries `__version__ = "0.0.1"`,
which has drifted by eight minor versions from `pyproject.toml`. Nothing
keeps the two literals in sync because nothing forces them to come from the
same place.

## Solution

Make the git tag the single source of truth. Remove the literal `version`
field from `pyproject.toml`, declare the version `dynamic`, and use
[`hatch-vcs`](https://github.com/ofek/hatch-vcs) as the build-time version
hook. `hatch-vcs` is the Hatch-native equivalent of `setuptools_scm`: it
derives the version from `git describe --tags` and writes a generated
`_version.py` into the package at build time.

This also resolves the concurrent-branch collision described above as a
direct property of the migration: because the version is derived at
build/tag-push time rather than being committed into the tree, two branches
no longer carry conflicting stamped versions at merge — neither stamps
anything.

## Changes

### `pyproject.toml`

Remove the static `version` field, declare it dynamic, add `hatch-vcs` to
the build requirements, and configure the hook.

```toml
[project]
name = "custom-sam-peft"
dynamic = ["version"]
# ... (rest unchanged: description, readme, license, etc.)

[build-system]
requires = ["hatchling", "hatch-vcs"]
build-backend = "hatchling.build"

[tool.hatch.version]
source = "vcs"

[tool.hatch.version.raw-options]
fallback_version = "0.0.0+unknown"

[tool.hatch.build.hooks.vcs]
version-file = "src/custom_sam_peft/_version.py"
```

`fallback_version` is the value used when no git history is available
(e.g., sdist builds, downloaded tarballs, build contexts without `.git/`).
It keeps `uv build` working in those environments rather than failing hard;
callers that want a real version in such contexts set
`HATCH_VCS_PRETEND_VERSION_FOR_CUSTOM_SAM_PEFT` (see Dockerfile section).

### `src/custom_sam_peft/__init__.py`

Replace the stale literal with a re-export of the generated module:

```python
"""custom_sam_peft — parameter-efficient finetuning of SAM3.1."""

from ._version import __version__

__all__ = ["__version__"]
```

`_version.py` is generated by the `hatch-vcs` build hook and carries the
real version string. It must not be committed (see `.gitignore`).

### `.gitignore`

Add the generated version file:

```
src/custom_sam_peft/_version.py
```

### `Dockerfile`

The build context does not include `.git/` — it is excluded by the
existing `.dockerignore`. Rather than un-ignoring it (which would balloon
the build context and invalidate Docker layer caching on every commit),
the image receives the version through an environment variable.

Use the project-scoped `HATCH_VCS_PRETEND_VERSION_FOR_<PROJECT>` form
rather than the global `HATCH_VCS_PRETEND_VERSION` so the override does not
bleed into any nested builds (e.g., the `sam3` direct-reference
dependency).

```dockerfile
# syntax=docker/dockerfile:1.7
FROM pytorch/pytorch:2.6.0-cuda12.4-cudnn9-runtime

RUN apt-get update && apt-get install -y --no-install-recommends \
      git \
      build-essential \
    && rm -rf /var/lib/apt/lists/*

COPY --from=ghcr.io/astral-sh/uv:0.5.11 /uv /uvx /usr/local/bin/

ARG HATCH_VCS_PRETEND_VERSION_FOR_CUSTOM_SAM_PEFT
ENV HATCH_VCS_PRETEND_VERSION_FOR_CUSTOM_SAM_PEFT=${HATCH_VCS_PRETEND_VERSION_FOR_CUSTOM_SAM_PEFT}

ENV UV_LINK_MODE=copy \
    UV_COMPILE_BYTECODE=1 \
    PYTHONUNBUFFERED=1 \
    HF_HOME=/workspace/.cache/huggingface

# ... (rest unchanged)
```

The `ARG`/`ENV` pair must appear **before** any `uv sync` step so the
build hook sees it when the project is installed. The existing `git`
install in the `apt-get` line is for fetching the `sam3` git+https
dependency; it is not used for version resolution, and no `.git/`
directory is present inside the image.

### `.github/workflows/docker.yml`

Both `docker/build-push-action` invocations need the pretend-version build
arg, derived from the pushed tag with the leading `v` stripped.

Add a step before the first build:

```yaml
- id: strip-v
  run: echo "version=${GITHUB_REF_NAME#v}" >> "$GITHUB_OUTPUT"
```

Pass it to both build-push steps:

```yaml
- uses: docker/build-push-action@10e90e3645eae34f1e60eeb005ba3a3d33f178e8
  with:
    context: .
    platforms: linux/amd64
    load: true
    push: false
    tags: ghcr.io/nguyenjus/custom-sam-peft:ci-smoke
    build-args: |
      HATCH_VCS_PRETEND_VERSION_FOR_CUSTOM_SAM_PEFT=${{ steps.strip-v.outputs.version }}
    cache-from: type=gha
    cache-to: type=gha,mode=max
```

```yaml
- uses: docker/build-push-action@10e90e3645eae34f1e60eeb005ba3a3d33f178e8
  with:
    context: .
    platforms: linux/amd64
    push: true
    tags: ${{ steps.meta.outputs.tags }}
    labels: ${{ steps.meta.outputs.labels }}
    build-args: |
      HATCH_VCS_PRETEND_VERSION_FOR_CUSTOM_SAM_PEFT=${{ steps.strip-v.outputs.version }}
    cache-from: type=gha
```

The workflow only triggers on `tags: ["v*"]`, so `github.ref_name` is
guaranteed to be a `v`-prefixed tag at this point. `actions/checkout`
needs no changes — the tag is already in `github.ref_name`, and the
workflow does not run `uv build` or `hatch build` directly.

### `.github/workflows/ci.yml`

`uv sync` and `uv lock --check` need to resolve a version. On PR builds
this will be a dev-version string like `0.8.1.devN+g<sha>` rather than a
release tag. For that to work, every `actions/checkout` step must fetch
full history and tags.

Apply to all four `actions/checkout` steps (jobs: `test`, `lock-check`,
`lint-hygiene`, `gpu-deselect-check`):

```yaml
- uses: actions/checkout@de0fac2e4500dabe0009e67214ff5f5447ce83dd
  with:
    fetch-depth: 0
    fetch-tags: true
```

`fetch-depth: 0` is required because `fetch-tags: true` alone still
operates against a shallow clone by default. The `lint-hygiene` and
`gpu-deselect-check` jobs do not strictly need tags today, but applying
the change uniformly keeps the workflow consistent and prevents future
surprises if those jobs ever start running `uv sync`.

### `tests/unit/test_version.py` (new)

One small test confirming the build hook produced a usable version:

```python
"""Smoke test that the hatch-vcs build hook populated __version__."""

from packaging.version import Version

import custom_sam_peft


def test_version_is_valid_pep440() -> None:
    assert isinstance(custom_sam_peft.__version__, str)
    assert custom_sam_peft.__version__, "__version__ must not be empty"
    # Raises InvalidVersion if not parseable.
    Version(custom_sam_peft.__version__)
```

`packaging` is a transitive dependency of `pip`/`uv`/`hatchling`, so it is
available in the test environment with no extra `pyproject.toml` entry.

### `uv.lock`

After the `pyproject.toml` changes, regenerate:

```
uv lock
```

The `lock-check` CI job (`uv lock --check`) will fail if the lockfile is
not updated.

## Verification

Run from a checkout with full git history:

- `uv build` produces a wheel; inspecting `wheel/METADATA` shows the
  expected version.
- Building from a clean checkout at exactly tag `vX.Y.Z` produces version
  `X.Y.Z` (no `.devN` suffix).
- Building from a non-tag commit produces `X.Y.(Z+1).devN+g<sha>`.
- `uv run python -c "import custom_sam_peft; print(custom_sam_peft.__version__)"`
  prints the expected version.
- `uv run pytest tests/unit/test_version.py` passes.
- `git status` shows `src/custom_sam_peft/_version.py` as untracked
  (created by the build) and not staged.
- On a tag-push, the Docker workflow builds successfully and
  `docker run --rm <image> pip show custom-sam-peft` reports the tag
  version with no leading `v`.

## Non-goals

- No changes to the public Python API.
- No new `--version` CLI flag (none exists today; can be added in a future
  PR).
- No changes to release cadence, branch protection, or tagging conventions
  beyond what is implied by removing the manifest literal.
- No changes for downstream consumers of `custom-sam-peft` — the wheel is
  installed the same way and exposes the same `__version__` attribute.

## Risks and mitigations

- **Builds outside a git checkout fail.** sdists, downloaded tarballs, or
  build contexts that omit `.git/` have no tag history to read.
  *Mitigation:* `fallback_version = "0.0.0+unknown"` keeps the build
  succeeding; the `HATCH_VCS_PRETEND_VERSION_FOR_CUSTOM_SAM_PEFT`
  environment variable (also accepted as a Docker build arg) overrides
  the fallback when the caller wants a real version.
- **Editable installs (`uv sync`) can show a stale `_version.py` after
  pulling new tags.** The version file is regenerated when build hooks
  run, not on every import.
  *Mitigation:* Documented `hatch-vcs` behavior; running `uv sync` again
  after fetching tags rebuilds the file. Not a blocker for production
  wheel or Docker builds.
- **Shallow clones in third-party CI lose tag history.** A consumer
  building from a shallow clone gets the `fallback_version`.
  *Mitigation:* `fallback_version` keeps the build green; the pretend-
  version env var is the documented escape hatch for consumers who need a
  specific version string.
- **`packaging` not available at test time.** The test imports
  `packaging.version.Version`.
  *Mitigation:* `packaging` is pulled in transitively by `uv`,
  `hatchling`, and `pip`. It is present in every realistic test
  environment for this project. If it is ever missing, the test fails
  loudly with an `ImportError` rather than silently passing.
